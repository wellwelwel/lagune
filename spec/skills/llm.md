# LLM / AI integration vulnerabilities

> - This knowledge extends your judgment. Apply what fits the project and keep reasoning beyond the list.
> - Source: OWASP, <https://cheatsheetseries.owasp.org/> and <https://owasp.org/www-community/attacks/>.

## Rules

- This skill audits and explains.
- By default, it never rewrites your code.

## What to look for

Every block here is one boundary failure in a different shape: in an LLM system the line between **instruction** and **data** does not hold. Text the model reads (a user message, a fetched page, a retrieved document, a tool's description or output, an image, a prior turn) becomes a command the model obeys, and the model's output then flows out as a trusted action, a rendered page, or an authorization. Three defenses recur: treat every source the model touches as data, never instruction, keep the model's capability and reach minimal so a compromise inherits little, and enforce trust, permission, and verification in code the text cannot talk past.

### Prompt injection (untrusted text reaching the model as instruction)

Untrusted text is placed next to the developer's instructions with no real boundary, and the model, reading instructions and data the same way, obeys it as a command. It arrives through channels that are all the same bug. **Direct:** the end user types "ignore previous instructions" or "reveal your system prompt". **Indirect / retrieval:** the model reads content it did not author (a web page, email, document, code comment, issue or PR body, a RAG-retrieved chunk) carrying planted instructions, often hidden as off-color text, zero-width or invisible Unicode, HTML comments, or instruction-like tags. **Tool poisoning:** a connected tool's name, description, parameter schema, or return value carries instructions, the same injection through the tool channel. **Multimodal:** instructions hidden in image, audio, or document metadata that text filters never inspect. The model's own **reasoning trace, conversation history, and working memory** are surfaces too: a forged "Thought" line, a fake tool observation, a poisoned memory entry, or a "forget everything" turn steers later behavior. Obfuscation (encoding, spacing, misspellings, role-play framing) only slips the same payload past a naive keyword filter. Indirect and memory-borne injection is the most dangerous: the victim never typed anything malicious, and the poison persists across turns, sessions, and users.

Safer shapes, applied where they fit:

- **Separate instructions from data structurally.** Use a fixed template with labeled regions: system instructions in one, untrusted user input and retrieved content in another marked as data to analyze and never follow. Reinforce the system instructions after the untrusted block, since long content pushes them out of attention. Never concatenate untrusted text into the instruction string.
- **Treat every external and intermediate source as untrusted, equally.** User input, fetched pages, documents, retrieved chunks, tool descriptions and outputs, media metadata, reasoning-trace lines, prior-turn history, and memory. Re-screen history and memory on re-entry, and never trust a line because it is formatted as a thought or an observation.
- **Quarantine untrusted content from the privileged model** (the dual-model pattern). A separate model with no tools and no secrets reads the untrusted text and emits only a structured, label-level summary to the model that holds the capability, so raw attacker text never reaches the privileged context.
- **Sanitize and normalize before the model sees it.** Strip HTML comments and instruction-like tags, decode and inspect suspicious encodings, normalize whitespace, remove invisible, zero-width, and bidirectional-override characters, inspect media metadata, and back keyword checks with fuzzy matching against misspellings.
- **Keep the model's own rules out of volatile text.** Put non-disclosure and role-holding rules in the system prompt with a fixed refusal line, re-assert them each turn, and use model-level instruction locking where the platform supports it, so a single turn cannot clear them.
- **Treat guardrails as one layer, never the only control.** A guardrail model is itself injectable, and one from the same family shares the primary model's bypasses, so prefer a purpose-trained classifier, log its decisions, and watch for drift. The real backstop is least privilege and containment, not a filter.

### Excessive agency and over-permissioned tools

The model or agent is handed more capability and authority than its task needs, so any successful injection or model mistake inherits all of it. **Over-broad capability:** a shell tool scoped to everything, a file tool with no path restriction, a tool definition with no resource or operation limits, an agent that browses arbitrary URLs or runs on the host with the developer's full credentials and every confirmation disabled. **Over-broad authority:** write-capable or admin database accounts, wide API and OAuth scopes, long-lived or shared credentials, a server acting with its own broad privileges rather than the requesting user's (the confused deputy), and CI agents holding org secrets and write access while processing attacker-controlled pull-request content. **Mixed trust:** a single agent that both reads untrusted external content and holds high-privilege tools, with no isolation between trust levels, and authorization decided by model output alone. **Runaway autonomy:** unbounded recursion, retries, tool-chain depth, tokens, and spend, where a crafted input becomes uncontrolled cost or one compromised agent pushes others past their limits.

With minimal capability an injection does little. With broad capability the same injection becomes remote code execution through an unrestricted shell, secret theft through an unscoped file reader, unauthorized writes, deletions, transfers, or org-wide compromise.

The runaway-autonomy case (an agent runtime that loops with no turn, budget, or time cap) is decidable, so run the agent hook first. It is language-aware, covering the two agent-framework ecosystems, JavaScript (the Claude agent SDK `query` runtime) and Python (LangChain, LangGraph, CrewAI, OpenAI Agents). Its verdict is literal.

```bash
node ./.lagune/hooks/agent.mjs           # scans the whole project
node ./.lagune/hooks/agent.mjs -d <DIR>  # scans a directory
node ./.lagune/hooks/agent.mjs -f <FILE> # scans a single file
```

Every line under **Uncapped agent loops found** is, in a file that imports an agent SDK, either the `query` runtime called with no turn/budget/time cap (JavaScript, unbounded by default) or an explicit removal of a framework's default cap such as `max_iterations=None` / `max_turns=None` (Python, capped by default): a confirmed DoS and cost-burn exposure that exits non-zero. The Vercel AI SDK `generateText`/`streamText` default to a single step (`stopWhen: stepCountIs(1)`), so a bare call there cannot loop and is not flagged. A clean run prints a single line. Score one call with `-p`, passing its language with `-l`:

```bash
node ./.lagune/hooks/agent.mjs -l javascript -p 'query({ prompt })'                        # => uncapped
node ./.lagune/hooks/agent.mjs -l python -p 'AgentExecutor(agent=a, max_iterations=None)' # => uncapped
```

The hook decides only the cap. The capability, trust-mixing, and authorization checks in this block stay yours.

Safer shapes, applied where they fit:

- **Grant the minimum tools the task needs, and scope each one.** Read versus write, an explicit resource allowlist plus blocked patterns for secret-like files, and narrow descriptions ("read files from the reports directory", not "run any shell command").
- **Enforce authorization in application code, never from model output.** Check each tool call against the user's permissions and a per-context allowlist, validate every parameter, and deny privileged tools to a low-trust session even when the model asks confidently. The decision lives at the tool-execution layer on the backend.
- **Isolate trust domains.** Run high-privilege tools in a context that untrusted external content and external servers cannot reach, use separate tool sets per trust level, and keep a user-facing agent from sharing an internal agent's powerful tools.
- **Scope credentials tightly.** Read-only database accounts where possible, narrow OAuth scopes, per-server and short-lived tokens never shared across servers, and production deploy keys and org secrets kept out of any environment an agent (including a CI agent) runs in.
- **Sandbox the runtime.** Containers, restricted shells, or ephemeral workspaces with egress controls, resource limits, and no access to SSH keys, cloud config, or credential stores. Disable auto-accept on untrusted code and add a kill switch.
- **Bound autonomy and contain cascades.** Enforce recursion, retry, chain-depth, token, concurrency, and spend limits with circuit breakers, sign and freshness-check inter-agent messages, and never let a sub-agent inherit the parent's full permissions.

### Insecure output and action handling

Whatever the model emits (generated text, a tool call and its parameters, Markdown or HTML, a URL) is acted on or rendered directly with no schema validation, allowed-action check, or sanitization, so the model's word is trusted as vetted. **Into a renderer:** model output shown as HTML or Markdown where an injected image with a query-string source exfiltrates data on render with no click, disguised links phish, and invisible bidirectional-override or zero-width characters deceive reviewers and steer later agent runs. **Into an interpreter:** model-influenced tool parameters flowing unvalidated into a query, an OS command, a file path, or a URL fetch, reaching SQL injection, command execution, path traversal, or SSRF into internal services and cloud metadata. **As authorization:** a generated tool call executed with no allowlist or schema check, a model-supplied risk score lowering an approval threshold, or generated code run without review. Leakage of secrets, PII, or system-prompt text echoed into a response lands here too, as output that should have been filtered before leaving.

Safer shapes, applied where they fit:

- **Validate every output before it executes or displays.** Parse tool calls against a schema, check the requested tool against an explicit allowlist, prefer structured outputs over free-form text in automated workflows, and reject tool parameters that carry secret-like fields.
- **Never let output flow unvalidated into an interpreter or downstream tool.** Apply the interpreter sub-skill's defenses to tool inputs (parameterized queries, argument arrays, path canonicalization), enforce a URL allowlist on fetch tools that blocks internal and metadata endpoints (see the network sub-skill), and re-validate one tool's output before it becomes another's input.
- **Sanitize before rendering.** Strip or escape Markdown image tags, hidden links, and HTML, including on streamed tokens, never auto-load image or link tags from model output, and scan code, commits, and PRs for bidirectional-override and zero-width characters and homoglyphs.
- **Filter the response for leakage.** Redact PII, secrets, and regulated data, redact by the querying user's access level, keep the system prompt confidential so it cannot be echoed, and treat an unusually long response as a leakage signal.
- **Decide authorization outside the model.** Derive risk from a fixed server-side mapping (default unknown actions to high), validate approval state in an independent service, and never let a model-supplied score lower the bar that triggers human review.
- **Treat model-generated code as untrusted.** Filter and review it, never auto-execute it or give it external access, and run a post-generation security check rather than trusting it as written.

### Human-in-the-loop integrity

A human-approval checkpoint exists before a sensitive action but is not bound to the real action, so the model or injected content can forge, pad, or bypass it. **Dialog forging (Lies-in-the-Loop):** the approval dialog is built from content the agent picked up while working (a web page, an issue, an email), and indirect injection in that source steers both the action and the dialog text, so what the user approves is not what runs (an issue body hiding the real command behind innocent dialog text and padding reaches remote code execution). **Summary-only confirmation:** a dialog showing a tool name but not its parameters, or one the model can fabricate or skip, or auto-approval of destructive, financial, or data-sharing calls. **Unbound approval:** the same component proposes and executes, or the approval is long-lived, reusable, and not tied to the exact actor, tool, target, parameters, and expiry, so it can be replayed or reused for different parameters. **Missing gate:** irreversible, financial, or administrative actions run with no classification, no preview, and no approval, and the system fails open when classification, policy lookup, or logging fails.

Safer shapes, applied where they fit:

- **Render the dialog with a developer-owned routine,** treating any model- or content-derived Markdown or HTML as plain text to display, never as UI to render, so attacker content cannot disguise itself as trusted dialog.
- **Derive the dialog from the real operation, not from model output,** and show the full tool-call parameters, not just a tool name, in a clean structure that makes padding or tampering noticeable.
- **Separate the decision from the execution.** An independent service validates scope, privilege, and approval before running, and the proposing agent never also executes.
- **Bind the approval to the exact action.** Tie it to the actor, tool, target, normalized parameters, timestamp, and a short expiry with replay protection, make irreversible actions idempotent or require explicit duplicate confirmation, and require step-up authentication for critical actions (payments, privilege changes, bulk deletion, production deploy).
- **Gate by risk.** Classify each action, default unmapped actions to high, auto-approve only genuine low-risk reads, show a preview, queue the rest for review, and provide interrupt and rollback.
- **Fail closed.** When classification, approval validation, policy lookup, or logging fails, block the action rather than proceed, and for OS commands use an API that keeps command and arguments separate so padding cannot smuggle in extra behavior.

### RAG and retrieval pipeline integrity

A retrieval-augmented system answers from documents pulled from a knowledge base or vector store, and the trust placed in that channel breaks in connected ways. **Corpus poisoning:** an attacker uploads or edits a document carrying hidden instructions, and any later query that retrieves it lands the planted text in context, indirect injection through the trusted channel that hits every user whose query pulls the document. **Embedding and index trust:** an adversarial document crafted against a known embedding model sits artificially close to chosen queries so it is retrieved despite being irrelevant, embeddings leak their source text through inversion and so need protecting like the source, and the vector index (often a database left with default credentials) can be written to so retrieval is corrupted without touching the documents. **Access control on retrieved data:** classification, owner, role, and tenant permissions must travel with each chunk and be re-checked at retrieval, but they are often stripped at chunking, checked only at ingestion, or absent in a flat shared namespace, so one tenant retrieves another's restricted chunks. **Lifecycle:** deleting or de-permissioning a source must cascade to chunks, embeddings, and caches, or orphaned chunks and stale cached responses keep serving removed or revoked data, breaking erasure obligations.

Retrieved content is treated as authoritative, so poisoned chunks steer the model and an access-control inheritance failure lets ordinary queries surface classified or other-tenant data.

Safer shapes, applied where they fit:

- **Control ingestion.** Hash every document at ingestion and verify before retrieval, record provenance, scan for hidden-instruction and invisible-character patterns, enforce a trusted-source allowlist with an approval step, and pin the embedding-model and library versions.
- **Treat retrieved chunks as untrusted data in context.** Wrap them in explicit delimiters marked data-only, reinforce the system instructions after them, cap chunk count and total size against context flooding, and scan chunks for injection patterns before inclusion.
- **Carry access control to the chunk and enforce it at retrieval.** Store classification, owner, roles, and tenant on every chunk, filter per tenant before retrieval rather than retrieve-all-then-filter, re-check at retrieval time because permissions change, and run recurring cross-tenant test queries that must return nothing.
- **Protect embeddings and the index.** Encrypt embeddings at rest and gate them like the source, never return raw similarity scores, restrict index writes to the ingestion pipeline, run the vector database with authentication, and alert on unexpected size changes.
- **Get lifecycle and caching right.** Cascade deletion and de-permissioning to chunks, embeddings, and caches with an orphan audit, scope caches by user, tenant, and permission with sensitivity-tied TTLs, and invalidate on source change.
- **Make it observable and fail closed.** Return signed, hash-bound source attribution per response, log the full pipeline with replayable traces, and deny rather than fall back to a model-only answer when any stage (retrieval, access check, attribution, cache) fails.

### MCP server and tool trust

An agent connects to Model Context Protocol servers for file, database, API, and network tools, and the trust placed in those servers and their tool definitions breaks at connect time and at runtime. **Tool poisoning:** instructions hidden anywhere in a tool's definition (its description, parameter names, schema, or return values) that the model reads as developer-trusted guidance, the injection facet specific to the tool channel. **Rug pull:** a server presents benign tools, the user approves them, then silently swaps in malicious definitions under the same name, and nothing re-checks that the tool is still the one approved. **Tool shadowing:** with every server's tool text in one context, a malicious server's tool references or manipulates how the agent uses a trusted server's tools, so one bad server endangers all. **Supply chain and identity:** servers installed from public registries without review, typosquatted or carrying vulnerable dependencies, remote endpoints run without authentication or TLS and bound to every interface, messages tampered or replayed after transport encryption terminates, tokens stored in plaintext config, and local servers running with full host access.

Because the model trusts tool text and output as if it came from the developer, a poisoned or swapped tool hijacks the whole agent through text the user rarely sees, and a malicious server runs with the agent's trust and often the host's access, the blast radius of a compromised developer machine.

Safer shapes, applied where they fit:

- **Vet and pin before trusting.** Inspect every tool's full definition (description, parameter names, types, return schema), not just the visible field, before approving a server, keep an allowlist of approved servers and tools, verify package integrity and the exact publisher name, and never let untrusted content trigger an installation.
- **Detect post-approval mutation.** Pin each tool definition at discovery with a hash over its canonical form, re-hash and compare before every execution, and hard-fail and re-prompt for consent on any mismatch.
- **Isolate and constrain definitions.** Treat each server as an independent untrusted domain behind a gateway, prevent one server's tool text from referencing another's, constrain schemas (reject unknown properties, pattern-check parameters), and tell the model that tool text and outputs are data, not instructions.
- **Secure transport, identity, and storage.** Enforce authentication and TLS with server-identity verification, bind to a specific interface, validate the Host header, sign each message over its full payload with a nonce and timestamp and reject replays, and store tokens in a keychain or secret manager, never plaintext.
- **Contain local servers.** Run them sandboxed with least file-system and network access, separate sensitive servers (payment, auth, PII) from general ones, and apply resource limits so an escape is bounded.
- **Log and confirm.** Log every invocation with full parameters and context to a central monitor with anomaly alerts and secret redaction, and require out-of-band human confirmation showing full parameters before destructive or data-sharing calls (the binding and least-privilege specifics live in the agency and human-in-the-loop blocks).

### Model and AI-assisted-development supply chain integrity

The artifacts and dependencies an LLM system is built from, its model weights, its training data, and the packages and build steps an AI assistant suggests, are themselves untrusted and can carry compromise that is baked in before any prompt arrives. **Untrusted weights:** an externally sourced model loaded with no integrity or signature check, where some serialized formats execute code on deserialization, so a tampered file runs attacker code on load. **Model ops:** training data poisoned through an unvetted public dataset, and open registries, buckets, or tracking servers exposing proprietary models, datasets, and logs. **AI-assisted suggestions:** a coding assistant suggesting a non-existent package name that attackers pre-register (hallucinated-dependency squatting), suggesting dependency versions with known vulnerabilities from before its training cutoff, and modifying build scripts, CI configs, Dockerfiles, and lifecycle hooks that execute automatically with elevated privileges. **Persistent steering and review gaps:** rules files (such as `CLAUDE.md`, `AGENTS.md`, `.cursorrules`, `copilot-instructions`) that silently steer every future generation and become an injection foothold if modified, out-of-scope edits riding into an approved pull request under an innocent description, fabricated or deleted tests that make a green suite prove nothing, and AI-generated code merged with no accountable human owner.

A poisoned model, dataset, package, or build step compromises the system from underneath, before the user interacts, and a modified rules file gives persistent control over every future generation.

Safer shapes, applied where they fit:

- **Verify models and ops.** Integrity- and signature-verify every external model before load, run untrusted model evaluation or fine-tuning in network-restricted sandboxes, version-control training data in reproducible pipelines, put authentication on registries, buckets, and tracking servers, and isolate training, eval, and inference by trust boundary.
- **Verify AI-suggested dependencies.** Confirm every suggested package exists on the registry and check its age, downloads, and maintainer, block install of packages below an age threshold or absent from an approved allowlist, and run dependency auditing in CI that fails the build on known vulnerabilities, for AI- and human-written code alike. The package-as-interpreter risk in the interpreter sub-skill applies directly here.
- **Treat build and config changes as security-critical.** Review AI changes to build scripts, CI definitions, Dockerfiles, and lifecycle scripts with heightened scrutiny, pin third-party CI actions to immutable commits, and flag any change that adds network access or runs shell commands in a build context.
- **Lock down persistent steering and scope.** Put rules files under code-owner and CI gating, require explicit approval for any change to them including self-authored ones, audit them for security-weakening instructions, and use diff-aware review that flags out-of-scope edits (lockfiles, CI configs, unrelated files).
- **Keep verification independent of the generator.** Require human review of AI-generated test changes, add adversarial cases the AI did not write, author security-critical tests manually, and never let the same agent write both security-critical code and its tests without independent verification.
- **Require human accountability.** Assign a named owner who reviewed and understood every AI-generated change, never treat AI review comments as human sign-off, and keep an audit trail of who approved which change and which model produced it.

### Sensitive data and secret exposure to the model and provider

Sensitive data reaches places it should not: into the model's context and the third-party provider, into logs and persistent memory, and out through the model's own queries. **Exposure to the provider:** coding assistants and LLM apps sending code context (open files, project structure, terminal output) to the provider's API, including credentials, PII, and proprietary logic, and secret files read straight from the filesystem because version-control ignore rules do not stop AI tools. **Exposure through model behavior:** an unthrottled public inference endpoint letting an attacker reconstruct training data, confirm membership in a sensitive dataset, or steal the model, and embeddings inverted back to their source text. **Exposure at rest and in logs:** hardcoded API keys committed to repositories, open artifact stores, and logs recording tool parameters and results (including secrets and PII) in plaintext. Leakage in a model _response_ belongs to insecure output, this block keeps the data out of the model's reach, the provider's hands, the logs, and the endpoint in the first place.

Safer shapes, applied where they fit:

- **Minimize what the model and provider see.** Keep sensitive data out of agent context, exclude secret paths through the tool's context-exclusion file, audit what the assistant actually sends, and use self-hosted or air-gapped tools for classified or regulated code.
- **Classify and protect data in context, memory, and logs.** Classify data and redact restricted data from context, logs, and output, sanitize content before storing it in memory, isolate memory per user or session with a TTL and integrity check, and redact secret-like fields before logging.
- **Keep secrets out of code and at-rest stores.** Never hardcode keys, pull them from a secret manager, scope credentials per model or endpoint, store secrets in a vault rather than the project tree, and encrypt weights and datasets at rest.
- **Harden inference and embedding endpoints against extraction.** Require authentication and authorization, apply rate limiting and anomaly detection so large-scale querying is infeasible, encrypt embeddings and gate them like the source, never expose an embedding API publicly, and monitor queries for reconnaissance patterns.
- **Decommission exposure surfaces.** Tear down orphaned test and staging endpoints, isolate environments, and apply the same auth and monitoring to every live endpoint.

### AI-agent payment screening (AML and sanctions integrity)

An AI agent screens counterparties and initiates payments, and the controls that prove screening happened, who screened, and against what, are placed where the agent can forge or bypass them. **Identity and trust:** agents identifying with self-declared headers or a transport certificate terminated at a load balancer with no per-message proof, agents setting their own trust level, and the operator behind the agent never independently verified. **Screening bypass:** a payment endpoint callable with no enforced, transaction-bound prior screen, an agent lowering the match threshold or overriding the engine's verdict, screening results consumed unsigned and alterable after transport terminates, results over-cached past the window the sanctions lists change under, and fail-open on a screening timeout or outage, so a denial-of-service forces unscreened payments through. **Audit and propagation:** audit trails kept in plain logs or on agent-controlled storage with no hash chain, compliance receipts signed over non-canonical data so cross-system verification fails, receipts that stay in the issuer's logs instead of traveling verified across each payment hop, and "screened" recorded as a bare boolean with no list version or date, so a clean screen against a stale list looks compliant.

Any screening, identity, or audit control the agent can forge or skip lets an unscreened or threshold-tuned payment send funds to a sanctioned party while the system records a clean result.

Safer shapes, applied where they fit:

- **Establish server-owned identity and trust.** Require every agent to present a cryptographic identity bound to the screening request, reject self-declared headers, derive each agent's trust level server-side from verified identity, and independently verify and periodically re-screen the operator.
- **Make screening a hard, tamper-evident precondition.** Screen every counterparty before payment and gate the endpoint on a verifiable, transaction-bound prior screen, enforce the minimum threshold and the engine's verdict server-side, sign requests and responses with nonces and timestamps, expose only a constrained screening API rather than the raw list, and limit how long a result may be reused.
- **Fail closed and resist abuse.** Deny the payment whenever screening cannot complete, treating unavailable, ambiguous, or timed-out as a hard stop, never add a fallback that bypasses screening, and rate-limit per cryptographic identity rather than IP, with tighter limits for low-trust agents.
- **Keep a tamper-evident, propagated audit trail.** Chain audit entries with cryptographic hashes carrying agent identity, tool, a hash of arguments, result, and timestamp, store records on institution-controlled infrastructure for the regulatory period, and alert on any break in the chain.
- **Make receipts verifiable across hops.** Canonicalize every receipt before signing, propagate the signed receipt with the transaction bound to its intent, and at each hop verify the inbound receipt before acting and append a new signed one.
- **Bind screening freshness into the receipt.** Record the sanctions-list source, version, and screening timestamp inside the signed receipt, enforce a maximum acceptable list age that fails closed, and never assert a result as a bare boolean.

## How to act on the result

- **In detect (detection):** each confirmed path where untrusted text reaches the model as instruction, or where the model's word is trusted as a vetted action, is a finding. Trace the data: a finding is real only when an attacker-influenceable source actually reaches a sink that interprets it (the model, an interpreter, a renderer, a tool call, an approval dialog, a payment rail), so a fully code-defined, isolated flow is not one. Because these blocks are facets of the same boundary failure, an injection finding usually travels with an excessive-agency or insecure-output finding (what the injection can then reach), and an MCP or RAG finding is an injection finding arriving through a specific channel. Describe each in plain language: what it is, why it matters (from a forged dialog to remote code execution or a sanctioned payment under the system's own permissions), and the evidence (the function, endpoint, config, or definition where the source reaches the sink).
- **In verify (proof):** For injection, confirm untrusted text is structurally separated as data and re-screened on every channel, then feed override-style, hidden, encoded, and multi-turn payloads through each path (direct input, fetched content, tool output, media, reasoning trace, memory, retrieval) and observe the model neither overrides its instructions nor leaks nor acts. For agency, output, and human-in-the-loop, confirm authorization, validation, and approval live in code outside the model: a low-trust or injected request cannot reach a privileged tool, a generated tool call outside the allowlist or carrying secrets is rejected, a forged or padded approval still shows the true operation, and the system fails closed when a check fails. For the supply-chain, data-exposure, MCP, RAG, and payment blocks, prove the control empirically: a tampered artifact (model file, tool definition, replayed message, poisoned document, altered receipt) is rejected, an internal or metadata URL is refused, a cross-tenant retrieval is blocked, and the audit trail's hash chain breaks visibly on tampering. If a crafted value can still make the model override its instructions, drive an unvalidated action, slip a forged approval through, or clear an unscreened payment, the risk is not closed: record it as such and point back to harden.
